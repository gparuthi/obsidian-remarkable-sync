# 2026-07-01

## Per-page incremental sync: only write new/changed page images (GP-39)

Production issue: when a notebook's cloud mtime advances (one new handwritten page),
`processNotebook` re-rendered AND re-wrote ALL page JPEGs ‚Äî ~80 vault images
"modified" every day in the operator's vault. OCR already skipped unchanged pages
via `srcHash`; the image write did not. The cloud has NO per-page mtime (cPages =
id + idx only), so per-page change detection = the rendered-image hash.

### Change (branch `rmplugin/page-incremental`, stacked on `rmplugin/ocr-embed-top` / PR #8)

- `PageOcrState` gains two optional fields (backward compatible; legacy entries
  force one rewrite, then settle):
    - `imgHash` ‚Äî hash of the bytes actually written to the page's vault file.
      Only advances on a real write, so a "save images" off sync (which still
      advances `srcHash` for OCR) can't mark a stale file current.
    - `pageIndex` ‚Äî index the file was written at. The file NAME derives from the
      index while state is keyed by pageId; device page insert/delete/reorder
      shifts indexes, pointing a pageId at a DIFFERENT page's old file ‚Äî an index
      mismatch therefore forces a rewrite (hash equality alone would wrongly skip).
- New pure domain module `domain/page-image-sync.ts` (+spec):
    - `shouldWritePageImage(prev, srcHash, pageIndex, fileExists)` ‚Äî skip ONLY when
      the file exists AND `imgHash` matches the current render AND the index
      matches.
    - `pageImageState(prev, srcHash, pageIndex, wrote)` ‚Äî image facts to persist:
      written ‚Üí what's on disk now; not written ‚Üí carry prior facts forward.
    - `nonOcrPageState(...)` ‚Äî what a non-OCR sync persists: srcHash + preserved
      `ocrHash` when unchanged, `ocrHash: ''` (= must re-OCR) when changed.
    - `isPageOcrCurrent(prev, srcHash)` ‚Äî OCR skip now ALSO requires a non-empty
      `ocrHash`, so entries written by non-OCR syncs don't suppress OCR.
      Deliberately NOT a type predicate: false does not imply prev is undefined.
- Pipeline (`notebook-pipeline.service.ts`): hash computed once after `renderPage`
  (skipped entirely when neither saveImages nor OCR is on); `pagePath` built once
  per page and reused for the OCR figure embed; image write conditional; render
  failure keeps prior state and writes nothing (both modes) and is now counted
  (`renderFailed`); OCR failure paths keep prior OCR state but merge in fresh
  image facts (an image written this run isn't rewritten on retry). Final persist
  collapsed to a single `updateState` call (mtime held back only for
  OCR-incomplete runs; empty-map guard never clobbers existing state). Mid-loop
  incremental persist now merges over `prevPages` so a crash mid-run can't drop
  state for unreached pages.
- `markdown-writer.service.ts`: `vaultFileExists` via `Vault.getFileByPath` (+spec).
- Migration (`clean-image-placeholders.ts`): the pageId‚ÜínotebookId reconcile index
  now only includes pages with non-empty `ocrHash`, so the reconcile can't
  overwrite the `''` must-re-OCR sentinel (and the "OCR never ran ‚Üí skip vault
  scan" fast path still fires for non-OCR users).
- Sync-log per-notebook summary now includes `images: N new/changed, M unchanged`
  (+ `, K render-failed` when any) when "save images" is on; non-OCR summary
  status is `error` when a render failed.
- Business rules added to `documentation/Business Rules.md` (never rewrite
  unchanged page images; per-page state persisted by all syncs; `ocrHash ''`
  sentinel; `imgHash` advances only on real writes).

### /code-review round (high effort, 8 finder angles + adversarial verify)

Fixed (all verified CONFIRMED unless noted):

- **Index-shift wrong-image skip** (CONFIRMED): state keyed by pageId, path by
  pageIndex ‚Üí after a device page delete/insert/reorder the skip kept a different
  page's old image at the path. Fixed via `pageIndex` in state.
- **saveImages-toggle stale file** (CONFIRMED): srcHash advanced without a write ‚Üí
  re-enabling saveImages skipped a stale file forever. Fixed via `imgHash`.
- **Migration reconcile clobbers the `''` sentinel** (PLAUSIBLE): fixed via the
  non-empty-ocrHash index filter.
- **Mid-loop persist drops unreached pages on interrupt** (CONFIRMED,
  pre-existing): fixed via `{...prevPages, ...nextPages}` merge.
- False-success summary on non-OCR render failure; duplicated final-persist
  branches; triple `buildPagePath`; wasted hashing when nothing consumes it;
  missing `vaultFileExists` spec; Business Rules.md not updated.

Known/accepted (pre-existing, logged to builder-feedback for manager triage):

- Non-OCR mode never retries a render-failed page (mtime advances anyway) ‚Äî now at
  least visible in the log/summary.
- A sync with OCR toggled OFF advances the mtime of a partially-OCR'd notebook,
  parking its un-OCR'd pages until the device edits it again (recoverable via a
  manual single-notebook sync).
- Final persist drops per-page state for pages missing from this run's parse
  (transient parse failure ‚áí later one-time re-OCR/rewrite churn).
- A vault image whose CONTENT was corrupted/overwritten by a third party while
  hash+index still match is not repaired (existence is checked; bytes are not
  re-read) ‚Äî the pre-diff unconditional rewrite self-healed this.
- `rmdoc-import` still rewrites every page image (stateless one-shot import; the
  skip lives in the sync pipeline).

### Decisions

- No new setting ‚Äî unconditionally correct behavior.
- OCR path semantics unchanged (skip-unchanged, pacing, incremental persist,
  resume-until-allPagesOcrd).
- Out of scope: skipping download/render of unchanged `.rm` blobs via cloud index
  per-file hashes (separate issue later).

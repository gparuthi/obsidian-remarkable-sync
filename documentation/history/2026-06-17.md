# 2026-06-17

## Community-catalog review warnings: dependency vulnerabilities + filesystem disclosure

Addressed the warnings raised by the Obsidian community-plugin reviewer.

### Vulnerable dependencies — all dev-only, none bundled

Every flagged package (handlebars, lodash, tmp, minimatch, ajv, flatted, brace-expansion, picomatch, yaml, fast-uri, `@conventional-changelog/git-client`) is a **transitive devDependency** from build/release tooling (conventional-changelog-cli, eslint, commitlint, tailwind). The shipped `main.js` only bundles the runtime deps (`@leeoniya/ufuzzy`, `date-fns`, `immer`, `jszip`, `zod`) — none of which are flagged. So the vulnerabilities never reach end users. Build externals confirm only obsidian/electron/codemirror are external; runtime deps are bundled.

Fixed anyway to clear the warnings, via a `package.json` `overrides` block pinning patched versions:

- `handlebars` 4.7.9, `tmp` 0.2.7, `lodash` 4.18.1, `flatted` 3.4.2, `yaml` 2.9.0, `fast-uri` 3.1.2, `minimatch` 9.0.7, `brace-expansion` 5.0.6, `picomatch` 4.0.4, `@conventional-changelog/git-client` 2.7.0, `ajv` 8.20.0.

Decisions / gotchas:

- **Overrides only — no direct-dep bumps.** A full `bun update` was tried first but bumped `obsidian` → 1.13.1 (whose newer type defs demand `override` keywords → TS4114 errors) and pinned `zod` (a bundled runtime dep) to 4.4.3. Reverted; kept the change surface to overrides + lockfile so obsidian/zod stay put.
- **Bun does not support nested overrides** (warns and ignores them). So ajv could not be scoped per-parent. eslint needs ajv 6, commitlint needs ajv 8. Tested a flat `ajv: 8.20.0` override — eslint (flat config) lints clean under ajv 8 and all plugin rule schemas validate, so the flat override is safe here.
- `minimatch` 9.0.7 requires `brace-expansion` ^5, hence brace-expansion pinned to 5.0.6 (not 2.0.3).

### Direct filesystem access — kept, disclosed (user decision)

`src/app/services/auth/token-store.ts` uses `node:fs` to store reMarkable tokens at `~/.remarkable-sync/token.json`, outside the vault. This is a documented business rule (avoids accidental sync/sharing of long-lived credentials; reason the plugin is desktop-only). User chose to keep it and declare it rather than move tokens into vault plugin data. Strengthened the README **Privacy** section to state the rationale explicitly (only file touched outside the vault, why, desktop-only). Docs already disclosed the location (configuration.md, usage.md, tips.md).

### Verification

- `bun run tsc`, `bun run lint`, `bun test` (136 pass), `bun run build` all clean.
- Release tooling exercised under the bumped deps: `conventional-changelog` (git-client 2.7.0) generates, `commitlint` (ajv 8) loads its schema and applies rules correctly.
- Final dep scan: no flagged package remains in a vulnerable version range.
- `bun run format:check` had flagged 2 source files (`remarkable-cloud.service.ts`, `sync-protocol.ts`) — **pre-existing on HEAD** (committed lockfile already pins prettier 3.7.4), unrelated to the dep change. Fixed with `bun run format` in a separate `style` commit; `format:check` now clean.

# 2026-05-30

## Fixed: notebook list empty (#6)

reMarkable sync v3 `/sync/v3/files/{hash}` now requires an `rm-filename` HTTP header whose value matches the blob's logical name. Without it the server returns HTTP 400 (`{"message":"unexpected 'rm-filename' http header"}`), so every blob download (root index, per-document index, content files) silently failed and `listDocuments()` returned `[]`. Same root cause hit a sibling plugin (TimDommett/Remarkable-Sync — Obsidian, fixed in their 1.0.4 via #14).

### Changes

- `src/app/services/cloud/sync-protocol.ts`
    - `fetchBlob` now takes a required `rmFilename` and sends it as `rm-filename`.
    - Exported `ROOT_INDEX_FILENAME = "root.docSchema"` and `docIndexFilename(docId)` helpers.
- `src/app/services/cloud/remarkable-cloud.service.ts` — pass the correct filename at every call site:
    - Root index → `ROOT_INDEX_FILENAME`.
    - Per-entry / per-document index → `docIndexFilename(entryId | documentId)`.
    - Content blobs → real filename from the parsed index entry (`entry.id`, e.g. `<uuid>.metadata`, `<uuid>.content`).
- `src/app/services/cloud/sync-protocol.spec.ts` — new tests assert the `rm-filename` header is sent on `/files/{hash}` requests and that the filename helpers return the documented values. `fetchRootHash` (`/sync/v3/root`) was not affected — only the `/files/{hash}` endpoint requires the header.
